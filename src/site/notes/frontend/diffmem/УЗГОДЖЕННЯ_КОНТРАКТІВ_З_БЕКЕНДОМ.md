# Узгодження контрактів DiffMem з бекендом

> Джерело істини для контрактів: бекенд-пакет DiffMem у BLOOMback.
> Фронтенд не розширює API самостійно.

---

## 1. Канонічний принцип узгодження

Фронтенд приймає як незмінний інваріант:

- primary index містить тільки `present-state surface`
- history викликається лише `on demand`
- `depth` є частиною `context assembly`, а не search-поведінки
- history/diff/revision є окремими temporal routes
- `writer staging -> commit once` є бекенд-семантикою, яку фронтенд може лише відображати

---

## 2. Таблиця `бекенд ↔ фронтенд`

| Бекенд-термін / контракт | Канонічне значення | Фронтенд-проєкція |
|--------------------------|--------------------|-------------------|
| `depth=surface` | стислий current-state context | перемикач depth: режим швидкого перегляду |
| `depth=wide` | кілька релевантних сутностей без history | перемикач depth: режим широкого огляду |
| `depth=deep` | повний current-state context без history | перемикач depth: режим повного читання |
| `depth=temporal` | `deep` + history attachments on demand | перемикач depth: режим temporal inspection |
| `present-state search` | `GET /v1/memory/search` повертає тільки current-state entities | звичайний пошук entity, без history entries у списку результатів |
| `on-demand history` | history не індексується, а читається окремими routes | окрема history panel або temporal attachments |
| `entity history` | `GET /v1/memory/entities/:id/history` | список ревізій у панелі сутності |
| `entity diff` | `GET /v1/memory/entities/:id/diff?from=&to=` | окремий diff viewer між двома вибраними ревізіями |
| `revision snapshot` | `GET /v1/memory/entities/:id/revisions/:sha` | перегляд historical snapshot конкретної ревізії |
| `context assembly with depth` | `POST /v1/memory/context` з полем `depth` | фронтенд-запит із user-selected mode |
| `writer staging -> commit once` | багато `stage`, один фінальний `commit` | status/progress індикатор для staged changes і фінального commit |
| `history not in primary index` | history payload не searchable | history UI ніколи не змішується з main search list |

---

## 3. Контракти, які фронтенд може використовувати

### 3.1 Present-state search

Канонічний контракт:

`GET /v1/memory/search?q=<query>&limit=<n>`

Фронтенд трактує результат як список поточних сутностей. Якщо користувач хоче історію, UI не повинен перевикористовувати search result як history source.

### 3.2 Context with depth

Канонічний контракт:

`POST /v1/memory/context`

Ключове фронтенд-поле:

```json
{
  "query": "Що змінилося у проєкті bloom?",
  "depth": "temporal"
}
```

### 3.3 History routes

| Контракт | Призначення фронтенду |
|----------|-----------------------|
| `GET /v1/memory/entities/:id/history` | список ревізій для sidebar/panel |
| `GET /v1/memory/entities/:id/diff` | порівняння двох вибраних ревізій |
| `GET /v1/memory/entities/:id/revisions/:sha` | перегляд знімка конкретної ревізії |

### 3.4 Writer staging

Канонічна бекенд-модель допускає два шляхи:

- явне staging API
- або `stageOnly + sessionId` у payload запису

Фронтенд не обирає новий протокол самостійно. Він повинен відобразити той варіант, який реально реалізує бекенд.

---

## 4. Що фронтенду заборонено припускати

| Заборонене припущення | Чому |
|-----------------------|------|
| history можна підмішати в основний пошук | це ламає канонічний інваріант `present-state index only` |
| `temporal` є просто "більше search results" | temporal додає окремий history layer, а не ширший search |
| diff є частиною `Entity.content` | diff є окремим temporal payload |
| кожне writer-оновлення завершується commit | канонічна модель: `stage many -> commit once` |

---

## 5. Точки UX-зв'язку з бекендом

| Backend capability | Frontend surface |
|--------------------|------------------|
| `depth` policy | depth switcher |
| `entity history` | panel "Історія" |
| `entity diff` | compare revisions modal/view |
| `revision snapshot` | read-only historical entity view |
| temporal context attachments | attachments block під відповіддю |
| staging / commit semantics | writer status banner або progress state |

---

## 6. Висновок для фронтенду

Головне правило просте:

- search показує `що відомо зараз`
- history показує `як це змінювалося`

Якщо ці два шари зливаються в одному UI-потоці без явної межі, фронтенд перестає бути сумісним із канонічною DiffMem-моделлю.
