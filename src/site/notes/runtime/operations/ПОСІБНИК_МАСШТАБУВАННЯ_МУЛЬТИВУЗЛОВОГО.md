# BLOOM Runtime — Multi-Node Scaling Guide

> Створено: 2026-02-27
> Статус: Canonical
> Scope: Горизонтальне масштабування — додавання worker-нод до BLOOM Runtime

---

## Передумови

- BLOOM Runtime працює і доступний (наприклад `bloom.exodus.pp.ua` або `http://<IP>:5000`)
- Protocol V1 E2E верифікований (базова нода "alpine" online)
- Claude CLI встановлений на цільовій машині

---

## 1. Автоматичне встановлення (one-liner)

```bash
curl -sSL https://bloom.exodus.pp.ua/api/runtime/agent-install-script?node_id=$(hostname)&server_url=https://bloom.exodus.pp.ua | bash
```

Скрипт:
1. Клонує репозиторій
2. Встановлює залежності Python
3. Створює `.env.agent`
4. Реєструє worker у Runtime API
5. Створює systemd/OpenRC сервіс

---

## 2. Ручне встановлення

### 2.1 Клонування

```bash
git clone https://github.com/maxfraieho/membridge.git ~/membridge
cd ~/membridge
```

### 2.2 Конфігурація

```bash
cat > ~/.env.agent << 'EOF'
MEMBRIDGE_SERVER_URL=http://192.168.3.184:8000
MEMBRIDGE_AGENT_KEY=<secret>
MEMBRIDGE_NODE_ID=$(hostname)
MEMBRIDGE_AGENT_PORT=8001
BLOOM_RUNTIME_URL=https://bloom.exodus.pp.ua
RUNTIME_API_KEY=<secret>
MEMBRIDGE_HOOKS_BIN=~/membridge/hooks
EOF
```

### 2.3 Запуск агента

```bash
cd ~/membridge
chmod +x run-agent.sh
./run-agent.sh
```

> **Важливо:** Завжди використовуйте `run-agent.sh` замість прямого виклику `python3 agent/main.py`.
> Скрипт автоматично завантажує `.env.agent`, знаходить Python venv та додає NVM/Claude CLI у `$PATH`.

### 2.4 Systemd сервіс (Debian/Ubuntu/Raspbian)

```bash
cat > /etc/systemd/system/membridge-agent.service << 'EOF'
[Unit]
Description=BLOOM Node Agent
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=vokov
WorkingDirectory=/home/vokov/membridge
EnvironmentFile=/home/vokov/membridge/.env.agent
ExecStart=/home/vokov/membridge/run-agent.sh
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal
SyslogIdentifier=membridge-agent

[Install]
WantedBy=multi-user.target
EOF
sudo systemctl daemon-reload
sudo systemctl enable --now membridge-agent
```

### 2.5 OpenRC сервіс (Alpine)

```bash
cat > /etc/init.d/membridge-agent << 'EOF'
#!/sbin/openrc-run
name="membridge-agent"
command="/home/vokov/membridge/run-agent.sh"
command_user="vokov:vokov"
pidfile="/run/membridge-agent.pid"
command_background="yes"
output_log="/var/log/membridge-agent.log"
error_log="/var/log/membridge-agent-error.log"
EOF
chmod +x /etc/init.d/membridge-agent
rc-update add membridge-agent default
rc-service membridge-agent start
```

## 3. Верифікація

```bash
# 1. Health check агента
curl -s http://localhost:8001/health | jq .

# 2. Перевірити реєстрацію в Runtime
curl -s https://bloom.exodus.pp.ua/api/runtime/workers | jq '.[].node_id'

# 3. Claude CLI доступний
which claude && claude --version

# 4. Тестова задача
TASK_ID=$(curl -s -X POST https://bloom.exodus.pp.ua/api/runtime/llm-tasks \
  -H "Content-Type: application/json" \
  -d '{"context_id":"test","agent_slug":"writer","prompt":"Say hello from '$(hostname)'","desired_format":"text"}' \
  | jq -r .id)

curl -s -X POST https://bloom.exodus.pp.ua/api/runtime/llm-tasks/$TASK_ID/dispatch | jq .
```

## 5. pickWorker — балансування навантаження

При dispatch задачі Runtime автоматично обирає worker:

```
1. Фільтр: status="online" AND claude_cli=true AND active_leases < max_concurrency
2. Sticky: якщо є context_id → prefer worker що вже працював з цим контекстом
3. Sort: free_slots DESC → last_heartbeat DESC
4. Якщо workers = 0 → HTTP 503 "No available worker"
```

Це означає: додавання нод → автоматичне масштабування без змін конфігурації Runtime.

---

## 6. Тест паралельного dispatch

```bash
# Створити 3 задачі і dispatch одночасно
for i in 1 2 3; do
  TASK_ID=$(curl -s -X POST https://bloom.exodus.pp.ua/api/runtime/llm-tasks \
    -H "Content-Type: application/json" \
    -d "{\"context_id\":\"scale-test\",\"agent_slug\":\"writer\",\"prompt\":\"Task $i: describe your node\",\"desired_format\":\"text\"}" \
    | jq -r .id)
  curl -s -X POST https://bloom.exodus.pp.ua/api/runtime/llm-tasks/$TASK_ID/dispatch | jq .
  echo "Dispatched task $i: $TASK_ID"
done

# Через 60с перевірити результати
sleep 60
curl -s https://bloom.exodus.pp.ua/api/runtime/runs | jq '.[] | select(.context_id=="scale-test") | {id: .id[0:8], status, worker_id}'
```

## 8. Auto-sync ключів (НОВЕ — 2026-02-28)

При реєстрації агента (`POST /api/runtime/workers`), Runtime повертає актуальні ключі:
- `membridge_agent_key` — для аутентифікації агента
- `membridge_admin_key` — для heartbeat до Control Plane

Агент автоматично оновлює `.env.agent` та поточне середовище, якщо ключі відрізняються.

**Лог при успішній синхронізації:**
```
auto-synced MEMBRIDGE_AGENT_KEY from Runtime (was abc12345...)
auto-synced MEMBRIDGE_ADMIN_KEY from Runtime (was def67890...)
```

**Це означає:** після додавання нової ноди ключі синхронізуються автоматично — достатньо вказати лише `BLOOM_RUNTIME_URL` та `RUNTIME_API_KEY` у `.env.agent`.

## 9. Signal decoding у exit codes (НОВЕ — 2026-02-28)

Якщо Claude CLI завершується з exit code ≥ 128, агент автоматично декодує сигнал:

| Exit code | Signal | Типова причина |
|-----------|--------|----------------|
| 137 | SIGKILL (9) | OOM killer або `kill -9` |
| 143 | SIGTERM (15) | Graceful shutdown або timeout |
| 139 | SIGSEGV (11) | Segmentation fault |
| 134 | SIGABRT (6) | Abort (assertion failure) |

Це допомагає швидко діагностувати проблеми на edge-нодах з обмеженою пам'яттю.
