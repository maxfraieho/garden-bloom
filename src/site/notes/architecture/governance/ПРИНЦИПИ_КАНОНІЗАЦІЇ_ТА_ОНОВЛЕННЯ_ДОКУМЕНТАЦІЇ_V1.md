# Принципи канонізації та оновлення документації

> Дата: 2026-03-15
> Scope: Garden Bloom knowledge base
> Джерела: Diátaxis, docs-as-code, web research 2026-03-15 + BLOOM NotebookLM
> Статус: Canonical governance reference

---

## 0. Призначення документа

Цей документ фіксує результати дослідження сучасних практик документування (Крок 0 аудиту 2026-03-15) і визначає:
- які зовнішні практики ми беремо в роботу;
- які НЕ переносимо буквально, бо в проекті вже є свій канон;
- як зовнішні практики співвідносяться з правилами BLOOM;
- які правила застосовуються під час аудиту і канонізації.

---

## 1. Дистиляція зовнішніх практик

### 1.1 Diátaxis (чотири квадранти)

| Тип | Призначення | Аудиторія |
|-----|-------------|-----------|
| Tutorial | Навчальний маршрут для початківця | Нові розробники |
| How-to guide | Рішення конкретного завдання | Компетентний практик |
| Reference | Технічні факти для пошуку | Будь-хто |
| Explanation | Відповідь на "чому" | Архітектори, розробники |

**Що беремо:** Розмежування factual / runbook / explanation / spec вже де-факто присутнє в системі тегування (format:spec, format:guide, format:contract, format:audit). Diátaxis підтверджує правильність цього підходу і додає аргумент проти змішування типів в одному документі.

**Що НЕ переносимо буквально:** Diátaxis не вимагає рівно чотирьох теків. Наша ієрархія (`architecture/core/`, `operations/`, `runtime/operations/`) вже адаптована під конкретний контекст execution platform.

### 1.2 Docs-as-code

- Документація в Git поряд з кодом, PR-review, CI-валідація
- Оновлення docs — обов'язкова частина будь-якого feature commit

**Що беремо:** Аудит та оновлення через Codex (не вручну), commit + push як фінальний крок. Автоматична валідація wiki-links.

**Що НЕ переносимо буквально:** Ми не маємо CI pipeline для docs, тому автоматичну валідацію замінюємо ручним аудитом через Codex.

### 1.3 Frontmatter / metadata discipline

- Поля `status`, `format`, `tier`, `created`, `updated` — обов'язкові
- Поле `title` повинно відповідати змісту, не назві файлу
- Стан `deprecated` / `historical` — явно через frontmatter

**Що беремо:** Мінімальний набір тегів вже описано в `КАНОНІЧНА_СИСТЕМА_ТЕГУВАННЯ.md` та інваріанті I-T1. Доповнюємо правилом: `title:` у frontmatter має бути україномовним (якщо документ україномовний).

### 1.4 Redirects / aliases після перейменувань

- Після rename URL → зберегти aliases у frontmatter
- Або оновити всі вхідні посилання

**Що беремо:** Після кожного перейменування файлу — оновлення всіх wiki-links у базі + можливий запис `legacy_name:` у frontmatter для провенансу.

**Що НЕ переносимо буквально:** Obsidian Digital Garden не підтримує server-side redirects для wiki-links. Тому canonical стратегія: оновлення всіх посилань + legacy_name у frontmatter.

### 1.5 Information architecture / index pages

- Index/navigation docs структурують граф, а не дублюють зміст
- Кожен каталог потребує `_INDEX.md`
- Маршрути читання ведуть читача вперед

**Що беремо:** Правило orphan docs (I-S3) + правило мінімум 2 outbound / 1 inbound wiki-link (I-S5, I-S6) вже це забезпечують.

### 1.6 Link hygiene / selective linking

- Кожне посилання повинно служити навігаційній або пояснювальній меті
- Не link spam
- Orphan docs = структурна помилка

**Що беремо:** Підтверджує I-S2, I-S3, I-S5, I-S6. Правило: після rename оновлювати всі stale links, а не залишати broken references.

### 1.7 "Incorrect docs are worse than missing docs"

- Застаріла фактична документація шкодить більше, ніж відсутність
- Якщо документ не може бути актуальним — маркувати його historical або deprecated

**Що беремо:** Документи категорії B (factual, але розійшовся з кодом) мають отримати `status:needs-update`. Документи категорії C (planned як factual) мають `status:draft` або `status:planned`. Це вже частина аудиту.

### 1.8 Language consistency

- Централізований глосарій + затверджені переклади
- Нормалізація: не змішувати мови в поясненні

**Що беремо:** Повністю відповідає `КАНОНІЗАЦІЯ_МОВИ.md`. Доповнення: `title:` у frontmatter має бути мовою документа.

### 1.9 Розведення factual / plans / runbooks / historical

- Runbook: покроковий операційний сценарій
- Plan/proposal: майбутня дія, яка ще не відбулась
- Historical: архівний знімок, не оновлюється
- Factual: відповідає поточному стану системи

**Що беремо:** Класифікація A/B/C/D/E з аудиту відповідності (2026-03-15) + system tag taxonomy.

## 2. Правила BLOOM, що залишаються незмінними

Ці правила вже закріплені в канонічних governance docs і не змінюються:

| Правило | Документ |
|---------|----------|
| Українська — канонічна мова документації | [[КАНОНІЗАЦІЯ_МОВИ]] |
| MEMBRIDGE, BLOOM, DiffMem, DRAKON — дозволені виключення | [[МАТРИЦЯ_ТЕРМІНІВ_ТА_ЗАМІН]] |
| Теги: domain:*, status:*, format:* — обов'язкові | [[КАНОНІЧНА_СИСТЕМА_ТЕГУВАННЯ]] |
| Кожен doc ≥2 outbound wiki-links | [[ІНВАРІАНТИ_ГРАФУ_ЗНАНЬ]] I-S5 |
| Orphan doc без `isolated:intentional` — помилка | [[ІНВАРІАНТИ_ГРАФУ_ЗНАНЬ]] I-S3 |
| Tier 1 = canonical specs, Tier 2 = operational / derivative | [[ІНДЕКС]] §NotebookLM |

## 3. Правила, застосовані під час аудиту 2026-03-15

### Naming

1. Файли з англійськими назвами, що не є виключеннями (продуктові назви, технічні ідентифікатори, README, код), перейменовуються в `UPPER_SNAKE_CASE.md` українською.
2. Після перейменування: оновити всі wiki-links + `_INDEX.md` + додати `legacy_name:` у frontmatter.
3. Файли в `workflow/` залишаються в `lowercase-kebab-case` — навмисна конвенція.
4. Файли з префіксом `BLOOM_RUNTIME_*` — продуктова назва, залишаються.
5. `MEMBRIDGE` в назві файлу — залишається як технічний ідентифікатор.

### Дублікати

1. Якщо існують EN і UA версії одного документа — EN версія отримує `status:deprecated` + `deprecated_by:` + посилання на UA canonical.
2. Видалення фізичного файлу — тільки після перевірки відсутності вхідних посилань і підтвердження Q.

### Frontmatter

1. `title:` у frontmatter має відповідати мові документа (українська для UA docs).
2. `status:` обов'язковий: canonical | historical | draft | deprecated | planned.
3. `updated:` оновлюється при кожній значущій зміні.
4. `legacy_name:` додається після перейменування.

### Links

1. Після кожного rename — масовий grep по всій базі на старе ім'я → оновлення.
2. Broken wiki-links → fix або `isolated:intentional` якщо orphan навмисний.
3. Link spam не додаємо — тільки навігаційно необхідні посилання.

### Класифікація документів

| Код | Статус | Дія |
|-----|--------|-----|
| FACTUAL_CANONICAL | status:canonical | Нічого не змінювати |
| FACTUAL_NEEDS_UPDATE | status:needs-update | Додати тег, позначити в audit |
| PLANNED_PROPOSAL | status:planned або status:draft | Перемаркувати, не "виправляти" |
| RUNBOOK_OPERATIONAL | format:guide, status:canonical | Залишити |
| HISTORICAL_AUDIT | status:historical | Залишити як є, не актуалізувати |
| DUPLICATE_MERGE_CANDIDATE | status:deprecated | Deprecated marker + посилання |
| DEPRECATE | status:deprecated | Deprecated marker |
| ORPHAN_NEEDS_LINKING | - | Додати вхідне посилання або isolated:intentional |
