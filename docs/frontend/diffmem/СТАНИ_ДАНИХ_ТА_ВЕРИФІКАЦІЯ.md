# Стани даних та верифікація DiffMem UI

> Документ фіксує view-model інваріанти фронтенду та перевірки сумісності з канонічним бекендом.

---

## 1. View-model шари

Фронтенд повинен тримати окремо три типи даних:

| Шар | Джерело | Призначення |
|-----|---------|-------------|
| `searchState` | present-state search | пошук поточних сутностей |
| `contextState` | `context` з `depth` | відповідь і контекст для current/temporal mode |
| `historyState` | history/diff/revision routes | інспекція ревізій і змін |

Інваріант: `historyState` не повинен міксуватися в `searchState`.

---

## 2. Станові машини UI

### 2.1 Depth-driven context

| Стан | Опис |
|------|------|
| `idle` | depth ще не вибрано або запит не відправлено |
| `loading` | фронтенд чекає відповідь `context` |
| `resolved:surface|wide|deep` | отримано current-state context |
| `resolved:temporal` | отримано context з temporal attachments |
| `error` | сталася помилка API |

### 2.2 History panel

| Стан | Опис |
|------|------|
| `closed` | панель не відкрита |
| `loading-history` | завантажуються ревізії |
| `history-ready` | ревізії доступні |
| `loading-diff` | запитується diff |
| `diff-ready` | diff готовий до показу |
| `loading-snapshot` | запитується historical snapshot |
| `snapshot-ready` | historical snapshot показано |

### 2.3 Writer staging

| Стан | Опис |
|------|------|
| `draft` | зміни ще не staged |
| `staged` | зміни staged |
| `committing` | commit in progress |
| `committed` | commit завершено |
| `commit-error` | фінальний commit не вдався |

---

## 3. Верифікаційна матриця `бекенд ↔ фронтенд`

| Перевірка | Що має підтвердити |
|-----------|--------------------|
| `search` не повертає history entries | фронтенд не показує history у main results |
| `depth=surface` не має temporal attachments | фронтенд не рендерить history block у factual mode |
| `depth=temporal` має temporal attachments | фронтенд уміє окремо показати history-derived data |
| `entity history` має `sha`, `timestamp`, `message` | history panel має достатньо даних для списку ревізій |
| `entity diff` має `patch`, `additions`, `deletions` | diff viewer не вигадує власний формат |
| `revision snapshot` повертає historical entity | snapshot view не використовує current entity як сурогат |
| `writer staging -> commit once` | UI не маркує staged зміни як committed |

---

## 4. Перевірки UI/UX

### 4.1 Depth switcher

- користувач бачить поточний режим
- `temporal` явно позначений як режим з історією
- переключення режиму не ламає базовий flow запиту

### 4.2 History panel

- відкривається тільки в контексті entity
- ревізії мають зрозумілий порядок
- можна вибрати дві ревізії для compare

### 4.3 Diff view

- patch читається окремо від current-state content
- `additions` і `deletions` видно без скролу по patch

### 4.4 Temporal attachments

- з'являються лише для temporal answers
- прив'язані до конкретної entity або ревізії
- не маскуються під "звичайні sources"

### 4.5 Staging status

- є явна межа між `staged` і `committed`
- користувач бачить, коли commit ще не відбувся

---

## 5. Ризики для фронтенду

| Ризик | Симптом у UI |
|-------|--------------|
| history pollution | історичні записи з'являються в основному search list |
| temporal overreach | `temporal` стає default для всіх запитів |
| semantic blur | користувач не розуміє, де current state, а де history |
| commit blur | staged changes виглядають як остаточно збережені |

---

## 6. Критерії приймання фронтенду

- depth switcher працює як явний контролер режиму контексту
- history panel, diff viewer і snapshot view існують як окремі UI-поверхні
- temporal attachments не змішуються з primary search
- staging/commit status відображає канонічний бекенд-flow
